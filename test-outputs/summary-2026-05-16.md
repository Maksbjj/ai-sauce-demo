## Known Limitations / Flaky Points

| Issue                  | Severity | Details                                                                                                                                                                          |
| ---------------------- | -------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Twitter/X rebrand      | LOW      | The app links to `twitter.com/saucelabs` which redirects to `x.com/saucelabs`. Test asserts on `x.com` which is correct today but may need updating if the domain changes again. |
| External URL stability | MEDIUM   | Tests 2 (About), 5 (Twitter), 6 (Facebook), 7 (LinkedIn) depend on live external sites. Failures could be environmental (network, CDN, rate limits) rather than app regressions. |
| Popup blocking in CI   | MEDIUM   | Footer link tests use `page.waitForEvent('popup')`. If CI blocks popups or has strict CSP, these tests will timeout.                                                             |
| Product URL ID mapping | LOW      | Product `?id=N` query params are hardcoded from UI analysis. If the backend changes product IDs the data table in the test must be updated.                                      |

## Quality Assessment

- **Structure:** All tests follow project conventions — `Should` prefix, `test.step` wrapping, fixture-based setup, no manual page management.
- **Locators:** All new locators use semantic `getByRole()` where possible; CSS `data-test$=` suffix pattern used for multi-element product collections, which is stable.
- **Assertions:** Hard `expect` used for navigation/URL assertions as required by project rules. No misuse of `softExpect`.
- **Maintainability:** Product data table is centralised — one place to update if products change. Social link pattern is consistent and easy to extend.
